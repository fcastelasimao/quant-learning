  yfinance occasionally returns malformed data on the first download after
  opening a fresh terminal. The second run always succeeds because the cache
  is warm. Add a retry loop (up to 3 attempts, 3-second pause between) with
  a minimum row count validation to eliminate this intermittent failure:
  ```python
  import time
  for attempt in range(3):
      try:
          raw = yf.download(...)
          if len(prices) < 100:
              raise ValueError(f"Too few rows ({len(prices)})")
          return prices
      except Exception as e:
          if attempt < 2:
              time.sleep(3)
          else:
              raise
  ```

- [ ] **Fix `validation.py` docstring -- section label above `run_walk_forward`**
  The section comment still implies it may be a Pareto section from an older

- [ ] **`optimiser.py` -- `de_objective` called with already-normalised weights**
  After clipping, `best_score = de_objective(best_weights)` is called, but
  `de_objective` internally calls `w / w.sum()` again on already-summed weights.
  This is harmless (normalising a normalised vector is a no-op) but is misleading.
  Either pass `best_weights` directly to `_score_allocation` or document the
  idempotency clearly.

  Email or push notification to tell users it is time to rebalance.
